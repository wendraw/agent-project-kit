# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies
npm install

# Type-check without emitting
npm run typecheck

# Build CLI (output: cli/dist/)
npm run build

# Install commands/skills to all IDE tool directories (.claude, .cursor, etc.)
npm run install:tools

# Export Mermaid mindmaps to Feishu-compatible text
npm run mindmap:render
```

> **Source of truth rule:** Only edit `commands/`, `skills/`, `agents/`. The `.claude/`, `.cursor/`, `.agents/`, `.kiro/`, `.trae/`, `.windsurf/`, `.opencode/` directories are **generated** — never hand-edit them. After changes, run `npm run build && npm run install:tools` to regenerate.

## Architecture Overview

This is a **requirement-driven AI workflow toolkit** (`agentkit` CLI). It orchestrates a multi-agent pipeline from PRD → code delivery, storing all state in structured files rather than conversation memory.

### Three-Layer Model

```
Command Layer  →  commands/*.md          (user entry points, installed to IDE tools)
Agent Layer    →  agents/*.md            (role contracts: router, req-manager, design-manager, impl-executor, experience-depositor)
Skill Layer    →  skills/*/SKILL.md      (execution checklists invoked by agents)
```

### State Machine

Requirements flow through 8 states with 2 manual review gates:

```
draft → intake-reviewed → design → design-reviewed → implementing → validating → completed → archived
         ⏸ 人工评审                   ⏸ 人工评审
```

State is persisted in `requirements/in-progress/<req-id>/.session-state.json` for cross-session continuity.

### Requirement Packet (9 Standard Artifacts)

Each requirement lives under `requirements/in-progress/<req-id>/`:

| File | Phase | Description |
|------|-------|-------------|
| `00-intake.md` | intake | PRD summary + discovery notes |
| `01-test-cases.mmd` | intake | Mermaid test case mindmap |
| `02-technical-solution.md` | design | Architecture & approach |
| `03-api-design.yaml` | design | OpenAPI 3.1 spec (source of truth) |
| `04-ui-handoff.md` | design | UI/UX design spec |
| `05-task-breakdown.yaml` | breakdown | Work split by component |
| `06-task-assignment.yaml` | breakdown | Task → repo mapping + dependencies |
| `07-delivery.md` | implementing | PR links + validation status |
| `08-retro.md` | archive | Retrospective |
| `task-input-{repo}.md` | implementing | **Self-contained** per-repo task package for code-repo agents |

### Context Assets

`context/` stores shared, reusable knowledge that compounds across requirements:

- `rules/` — Runtime rules (context, risk, pattern, service)
- `decisions/` — ADRs
- `patterns/` — Reusable engineering patterns
- `playbooks/` — Operational handbooks
- `records/experience/` — Indexed experience deposits from past tasks
- `tech/services/` — Per-repo service context generated by `/load-service`
- `index.json` — Master retrieval index (searched before every phase via `experience-retrieve` skill)

### Phase → Agent → Skills Routing

Defined in `workflow/phase-skill-map.yaml`. The `phase-router` agent determines current state from missing artifacts + `.session-state.json`, then delegates to the responsible agent with the correct skill set:

| Phase | Agent | Key Skills |
|-------|-------|-----------|
| intake | requirement-manager | req-create, index-manager |
| design | design-manager | design-create, experience-retrieve |
| breakdown | design-manager | design-change |
| task-input-gen | implementation-executor | design-implementation, workspace-setup |
| completion | requirement-manager | requirement-completer, experience-index |
| archive | experience-depositor | requirement-archiver, knowledge-evolution |

### Knowledge Compounding Loop

```
Execute phase → deposit experience (experience-index)
                                       ↓
Next phase → experience-retrieve searches index.json
             → surfaces Risk warnings + matched patterns
             → reduces repeated investigation
```

Gate failures trigger `error-to-experience` skill, converting blockers into Risk records that surface as `⚠ RISK` warnings in future phases.

Hit counts drive promotion: Doc (≥5 hits) → Skill → Command (≥10 hits).

## Key Design Constraints

- This repo is a **context repository**, not a code repository. Agents here generate `task-input-{repo}.md` packages; actual coding happens in the linked code repos.
- `requirements/repo-links.yml` maps repo aliases (`canvas-fe`, `main-fe`, `backend`) to git URLs.
- Design phase is **blocked** until `context/tech/services/{repo}.md` exists for every repo in `repo-links.yml`. Run `/load-service` first.
- `03-api-design.yaml` (OpenAPI 3.1) is the contract source of truth — `03-api-design.md` is auto-generated from it.
- Context budget per phase: L0 ~500 tokens, L1 ~1000–1500 tokens, L2 ~1500–3000 tokens. Trim order: keep Risk entries → keep high-frequency → truncate rest.

## CLI Development

The CLI source is `cli/src/cli.ts` (single file, ~4700 lines). Build target is `cli/dist/cli.js` (ESM, Node 20+). Key types:

```typescript
type RequirementState = "draft" | "intake-reviewed" | "design" |
                        "design-reviewed" | "implementing" |
                        "validating" | "completed" | "archived"
```

Feishu PRD extraction requires env vars `AGENTKIT_FEISHU_APP_ID` + `AGENTKIT_FEISHU_APP_SECRET`.
